# Scout 双轨决策模型

## 1. 为什么不能只用 λ

项目动作并不都属于同一种问题：

- “验证新检索路线能否达到最低召回率”是在购买信息；
- “修复阻断发布的构建错误”是在恢复可用性；
- “补充 API 文档”是在完成交付；
- “给 CI 增加类型检查”是在建立复利护栏。

因此，Scout 不使用单一公式覆盖全部动作。

## 2. 严重度类别

排序器使用以下类别带宽：

1. `blocker`：当前无法构建、测试、运行、部署，或存在立即的数据/安全风险；
2. `quality`：质量盲区、回归风险、重复缺陷和静态检查缺口；
3. `process`：自动化、CI、可观测性、复盘和交付流程；
4. `feature`：用户明确需要的新能力；
5. `knowledge`：文档、运行手册、架构记录和知识交接。

类别不是价值判断。用户明确目标可提升某个 feature 的优先级，但不能掩盖直接阻断。

## 3. 降风险轨

### 定义

对 `derisk` 动作：

\[
\lambda_h = \frac{-\ln p}{T_{min}/60}
\]

- `p`：最小验证通过的概率；
- `T_min`：获得足够决策信息所需的分钟数；
- `λh`：每小时失败发现率启发式。

### 解释

λh 越高，表示越可能用更短时间发现路线不成立。它适合在同一严重度、同一目标下比较风险实验，不是 Shannon 信息熵，也不是整个项目的绝对价值。

### 估计规则

使用概率档位：

| 档位 | 含义 |
|---|---|
| 0.20 | 证据明显不利，仍值得做快速上限测试 |
| 0.40 | 偏不确定或高风险 |
| 0.60 | 略偏可行，但存在关键未知 |
| 0.80 | 多数证据支持，仅需验证边界 |
| 0.95 | 基本确定，通常不再属于高价值降风险任务 |

`minutes` 是最小实验，不是完整实现。例如“完成一个检索系统”可能需要两周，但“用 30 条真实查询做上限测试”可能只需 20 分钟。

### 约束

- `0 < p <= 1`；
- `minutes > 0`；
- 排序器将 p 钳制到 0.05–0.99 用于防止极端值支配排序，但会保留警告；
- λh 仅用于 `derisk`；
- 证据置信度低时，不能仅凭高 λh 排到有明确证据的动作之前。

## 4. 执行轨

对 `execute` 动作：

\[
valueRate = \frac{impact \times 60}{minutes}
\]

并结合：

- `impact`：1–5；
- `compounding`：0–2，表示一次修复是否持续减少未来成本；
- `confidence`：0–1，表示证据可靠度；
- `reversibility`：0–1，越可逆越适合作为下一步。

执行轨更看重可交付价值，而不是失败发现率。

## 5. 排序键

排序器按以下稳定键排序：

1. 类别带宽；
2. blocker 内优先恢复基本可用性；
3. 同类别下，具有必要性的 `derisk` 在大规模执行前；
4. derisk：`λh × impact × confidence`；
5. execute：`valueRate × confidence + compounding bonus`；
6. 更小、更可逆、更易验证；
7. 原始输入顺序作为最终稳定排序键。

## 6. 证据置信度

- 0.9–1.0：命令结果、机器可读配置、明确用户目标；
- 0.7–0.89：多个文件信号一致；
- 0.4–0.69：有限证据支持的推断；
- <0.4：通常不应进入前三，除非它本身就是“验证未知”的动作。

## 7. 成功条件和停止条件

每个建议必须可证伪：

- 成功条件：量化通过标准，例如“Recall@10 ≥ 80%”；
- 停止条件：何时停止继续投入，例如“作弊上限仍低于 65%”；
- 升级条件：何时从小实验进入完整实现；
- 回退路径：失败后尝试哪一类替代方案。

这使 Scout 的建议从“意见”变成“可执行决策协议”。
