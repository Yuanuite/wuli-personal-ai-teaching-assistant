# Scout 完整输出示例

## 示例 1：无向量 RAG 项目

```markdown
## Scout 建议

> 状态：检测到 Python + TypeScript 双栈；31 个测试文件，未发现真实查询评测集；Git 有 6 个未提交文件。  
> 模式：deep｜证据置信度：中

### 🥇 用 30 条真实查询做无向量召回上限测试
- **类别 / 模式**：quality / derisk
- **证据**：31 个测试文件主要覆盖模块行为；仓库中未发现 recall、benchmark 或 eval 数据集。
- **为什么现在做**：如果真实同义词和模糊查询召回不足，继续完善 Agent 编排不会解决检索上限问题。
- **最小行动**：人工整理 30 条真实问题和期望文档，比较当前方案与 BM25/FTS 基线的 Recall@10。
- **成功条件**：当前方案 Recall@10 ≥ 80%，且不低于基线。
- **停止条件**：作弊式扩展关键词后仍低于 65%，转向混合检索或可选 embedding。
- **估计**：p=40%，20 分钟，λ=2.75/hour
- **影响**：5/5｜置信度：中

### 🥈 为检索评测建立可重复回归入口
- **类别 / 模式**：process / execute
- **证据**：未发现 eval 命令和固定查询集；目前结果无法跨版本比较。
- **为什么现在做**：一次建立后，每次策略修改都能自动判断召回是否退化。
- **最小行动**：增加 JSONL 查询集和单命令评测脚本，输出 Recall@5/10 与失败样例。
- **成功条件**：本地单命令运行并生成机器可读报告。
- **停止条件**：先完成优先级 1，若路线不可行则不扩展完整评测框架。
- **估计**：45 分钟｜影响/成本：高｜复利：2/2
- **影响**：5/5｜置信度：高

### 🥉 清理 6 个未提交文件并拆分可复核变更
- **类别 / 模式**：quality / execute
- **证据**：Git 工作区有 6 个未提交文件。
- **为什么现在做**：评测前需要知道结果来自哪一组变更，避免混合修改污染结论。
- **最小行动**：按“检索逻辑 / UI / 文档”分组审查差异，不自动提交。
- **成功条件**：每组变更目的清晰，可独立回退。
- **停止条件**：发现用户仍在编辑时，只记录分组建议，不改动文件。
- **估计**：15 分钟｜影响/成本：中｜复利：1/2
- **影响**：3/5｜置信度：高

### 其他观察
- 测试和构建入口仅被识别，未执行仓库代码。
- “未发现真实查询集”表示在受限扫描范围内未发现，不等于绝对不存在。
```

## 示例 2：项目很干净

```markdown
## Scout 建议

> 状态：Git 工作区干净；142 个测试文件；检测到单元测试和 CI，但未发现 E2E 目录。  
> 模式：standard｜证据置信度：高

### 🥇 验证一条核心用户路径是否被端到端覆盖
- **类别 / 模式**：quality / derisk
- **证据**：142 个测试文件，未发现 e2e、playwright、cypress 或 selenium 配置。
- **为什么现在做**：大量单元测试不能证明部署后的核心路径可用，先验证盲区是否真实存在。
- **最小行动**：选择登录后最关键的一条路径，在测试环境手工走通并记录断点。
- **成功条件**：路径无未覆盖的跨服务失败；若有，形成一个最小 E2E 用例。
- **停止条件**：确认已有等价外部监控或测试后，不重复建设。
- **估计**：p=80%，15 分钟，λ=0.89/hour
- **影响**：4/5｜置信度：中

### 🥈 补充发布与回滚运行手册
- **类别 / 模式**：knowledge / execute
- **证据**：README 存在，但未发现发布、回滚或 incident 文档。
- **为什么现在做**：项目质量已较稳定，当前最大风险转为交接和故障恢复速度。
- **最小行动**：记录发布命令来源、健康检查、回滚条件和负责人。
- **成功条件**：新成员仅凭文档可完成演练。
- **停止条件**：若已有外部手册，改为链接并验证其时效性。
- **估计**：30 分钟｜影响/成本：中｜复利：2/2
- **影响**：3/5｜置信度：中

### 其他观察
- 当前没有证据支持“存在阻断性 Bug”；Scout 不会为了凑满三条而制造问题。
```
