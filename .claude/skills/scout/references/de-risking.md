# 降风险技巧 —— 来自 Steinhardt「Research as a Stochastic Decision Process」

## 核心原则

**不要混淆「做任务」和「获取关于任务的信息」。**

目标不是完成任务，而是以最低成本判断任务是否可行。
降风险足够彻底之后，再切到执行模式全速推进。

## 具体技巧

### 上限测试（Ceiling Test）
先做一个"作弊版本"——比如直接看答案、用部署时拿不到的信息、手工模拟某个组件。
- 如果作弊版都跑不通 → 整个方向可以放弃，省下实现成本
- 如果作弊版跑通了 → 证明只要把各组件做好，系统就能成立

**什么时候用**：系统有多个耦合组件，不确定整体方案是否可行。

### 基线测试（Baseline）
用一个简单/现成的方法快速给出可达精度的下界。
- 如果复杂方法打不过简单基线 → 方向有问题
- 上限和基线一起，圈出可能的性能区间

**什么时候用**：实现了一个复杂方案，不确定是否真的比简单方法好。

### 暴力解先行
如果问题既有一个容易实现的暴力解，又有一个难实现的高效解：
- 先做暴力解，验证想法是否正确
- 暴力解也能用来调试高效解（对比输出）

**什么时候用**：高效解法是项目的主要不确定性，但暴力解法是例行的。

### 概念层剪枝（Concept-Level Pruning）
一次实现失败后，不要只记"X 不行"，要问"X 代表的这一类方案为什么不行"。
- 数学例子：不是"这个矩母函数界不够紧"，而是"矩母函数对我关心的分布根本给不出足够的控制"→ 排除所有矩母函数类方法
- 工程例子：不是"这个库的 API 不能用"，而是"这类库的设计假设和我们场景不兼容"→ 排除所有同类库

**什么时候用**：每次失败后。这是最高杠杆的习惯。

### 最小可行测试（Smallest Viable Test）
数据准备很耗时且可能失败，但你可以先下载几千条样本做实验。
- 目的是消除"数据是否合适"的不确定性，而不是"搞定数据"
- 在降风险模式下，10 分钟得到 80% 信心的测试比 2 小时 100% 信心的测试更有价值

**什么时候用**：任何耗时长的任务开始前。问自己："要判断这步是否可行，我能做的最小实验是什么？"
