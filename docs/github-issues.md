## 一、当前限制（6 个，label: enhancement）

### Issue 1: 检索：BM25 词法搜索无法捕获物理概念关系

**Labels:** `enhancement`, `knowledge-base`

**Body:**

当前 `kb.py` 使用 BM25 + 二元分词（bigram）做知识库检索。搜索"碰撞问题"找不到标记为"动量守恒"的条目，尽管它们属于同一物理领域。

**具体表现：**
- 扁平 JSON 索引没有概念层级（力学 → 动量 → 碰撞）
- 没有同义词扩展
- 没有跨概念关联
- 无法按知识领域范围搜索

**影响：** 平台的核心价值主张是知识检索和薄弱点分析，这是当前架构最大的单一能力缺口。

**建议方向：**
1. 构建高中物理概念本体（~200 节点），定义层级/关联/前置关系
2. 在 `kb.py` 的 `search()` 中基于本体做查询扩展
3. 可选集成小嵌入模型（multilingual-e5-small, 118M）做语义匹配

### Issue 2: 审核：教师审核是瓶颈，缺少风险分流与聚焦

**Labels:** `enhancement`, `teacher-console`

**Body:**

当前每道 AI 生成的答案都需要教师完整审核，即使是对 AI 从未出错的简单运动学题也是如此。审核负担可能导致"橡皮图章"效应。

**具体表现：**
- 所有答案无论难度和 AI 置信度都走相同的审核路径
- 没有风险分流（不同风险的题目仍使用相同展示密度和复核顺序）
- 审计记录是隐式的（JSON 时间戳和摘要哈希），不够结构化
- 没有审核疲劳检测机制

**影响：** 教师时间是乡村课堂最稀缺的资源，审核瓶颈降低了系统吞吐量。

**建议方向：**
1. 新增风险与置信度字段到独立验证报告，不让生成模型自评直接进入批准记录
2. 用风险等级调整队列顺序、突出可疑步骤并为低风险题提供精简复核视图，但始终保留教师显式确认
3. 结构化审计记录：时间戳、模型版本、prompt 哈希、编辑 diff、拒绝原因
4. 可选：合成错误注入（~1%）验证审核注意力

### Issue 3: 离线：缺少开箱即用的本地 SLM 配置与发现

**Labels:** `enhancement`, `agent-gateway`

**Body:**

当前 `agent_gateway.py` 已支持本机 JSON adapter 和回环地址的 OpenAI-compatible 服务，因此可以连接本地 SLM；但项目尚未自动发现 Ollama、检查本地模型是否已安装，也没有面向教师的一键配置。未预先配置本地 provider 时，断网仍会使答案生成与修改不可用。

**具体表现：**
- 没有 Ollama 等本地运行时的自动发现和模型就绪检查
- 本地服务需要教师手工填写端点与模型名，弱网现场不易排障
- 即使有网络，将学生作业发送到云端 API 也可能存在隐私顾虑

**影响：** 对没有网络的乡村课堂，系统从"降级可用"变成"完全不可用"。

**建议方向：**
1. 复用现有 OpenAI-compatible/JSON adapter 传输层，增加 Ollama 发现、模型清单与主动探测，不新增重复协议
2. 推荐模型：Qwen2.5-3B-Instruct（中文+物理强）或 Phi-4-mini（推理强）
3. 本机回环 provider 配置成功后优先于通用远程 CLI，并在 health 中明确标记 `data_locality=local`
4. UI 显示“本地模型生成，仍需教师复核”，支持稍后用其他 provider 重新运行

### Issue 4: 复习：间隔复习使用固定周期，缺乏自适应调度

**Labels:** `enhancement`, `knowledge-base`

**Body:**

当前 `kb.py` 的复习调度使用固定间隔（掌握度 0-5 映射到 1/3/7/14/30/60 天），不根据题目难度或学生表现自适应调整。

**具体表现：**
- 简单概念和学生反复答对的难题使用相同的复习节奏
- 反复答错的概念不会获得更高频的复习
- 不区分前提性概念（如 N3L）和孤立知识点的遗忘曲线差异
- 没有每道题的稳定性/难度参数模型

**影响：** 复习效率低，学生已经在错误类型上表现出明显差异但系统无法响应。

**建议方向：** 采用 FSRS v6 算法替换固定间隔（纯算法，~80 行 Python，Apache 2.0）：
- 每道题跟踪三个参数：稳定性、难度、可提取性
- 根据每次复习结果（正确/部分正确/错误）更新参数
- 自动计算下次最优复习时间
- 误差对比：SM-2 35.3 vs FSRS 19.9（越低越好）

### Issue 5: 工具接口：Skill 编排使用临时函数调用，缺少标准化工具接口

**Labels:** `enhancement`, `teacher-console`

**Body:**

当前 Skill 编排使用临时 Python 函数调用，接口不统一。新增能力（如 Windows OCR 引擎、电路仿真器）需要修改编排代码。

**具体表现：**
- 没有标准化的工具发现机制（有哪些可用能力？）
- 每个工具自行校验输入，没有统一的 JSON Schema 校验
- 没有结构化的错误分类
- 测试单个工具需要跑完整管道

**影响：** 系统的可扩展性和可测试性受限，跨学科扩展（数学、化学）时成本高。

**建议方向：** 采用 MCP（Model Context Protocol）stdio 传输模式：
- 将 OCR、知识库、物理仿真、答案生成包装为 MCP Server
- 使用 Python FastMCP 库的 `@mcp.tool()` 装饰器
- stdio 传输 = 纯子进程管理，零网络配置
- 获得自动 JSON Schema 校验、工具注解（readOnlyHint/destructiveHint）

### Issue 6: 验证：答案验证仅检查结构，缺少语义正确性校验

**Labels:** `enhancement`, `agent-gateway`

**Body:**

当前 `kb.py` 的 `validate_entry` 只做结构性校验（必要的标题存在、无 `[待核对]` 标记、图片引用可解析），无法检测物理推理错误、计算错误或概念混淆。

**具体表现：**
- 不验证答案步骤中的数字是否导向声明的结论
- 不检查推理链是否使用了正确的物理公式
- 不确认解答是否覆盖了题目的所有小问
- 结构完全正确但物理上错误的答案会通过校验

**影响：** 教师承担了全部语义审核工作。可能导致错误答案被标记为"通过"。

**建议方向：** 采用 Compound AI Systems 的 Generator-Verifier 模式：
- 生成步骤完成后，用一个独立的验证器模型检查答案
- 验证器检查：数学一致性、物理概念有效性、结构完整性
- 产出结构化报告（通过/失败/不确定 + 具体问题列表）
- 验证器可用更便宜的模型（验证比生成容易）

## 二、架构演进方向（8 个，label: enhancement）

### Issue 7: [架构演进] HITL 风险分流 + 结构化审计轨迹

**Labels:** `enhancement`, `architecture`, `teacher-console`

**Body:**

将现有四门禁审核系统形式化为：风险分级队列 + 结构化机器可读审计记录 + 保留教师最终确认的聚焦复核。

**解决的问题：** 教师审核瓶颈（见 Issue 2）

**方案要点：**
1. 在独立验证报告中记录 `risk_level`、`confidence` 与证据来源，不让模型自评直接改变批准状态
2. 高风险题优先排队并展开完整证据，低风险题使用精简复核视图；所有题仍由教师显式确认
3. 结构化审计记录：时间戳、模型版本、prompt 哈希、置信度、审核人身份、编辑 diff、拒绝原因
4. 可选：合成错误注入（~1%）验证审核注意力，防止疲劳效应
5. 风险阈值可按题目类型配置，但不得把阈值配置成自动执行 `approve-*`

**迁移复杂度：** 低 — 直接扩展现有审批门禁，不需要改状态机或 agent gateway

**投入产出比：** 高

**优先级：** P1；触发条件：先形成独立验证报告，并积累足以校准风险阈值的教师复核样本

### Issue 8: [架构演进] 混合语义搜索 + 物理概念本体

**Labels:** `enhancement`, `architecture`, `knowledge-base`

**Body:**

用预构建的高中物理概念本体 + 可选轻量级稠密检索来增强当前的 BM25 词法搜索。

**解决的问题：** BM25 无法捕获物理概念关系（见 Issue 1）

**方案要点：**
1. 创建 `physics-concept-ontology.json`（~200 节点），覆盖高中物理课程。定义 broader/narrower、related、prerequisite 关系
2. 在 `kb.py` 的 `search()` 中做查询扩展：每个查询词查找本体同义词和子概念，带衰减因子（0.7 权重）加入扩展词
3. 支持按本体分支范围过滤：`力学/动量` 返回所有动量相关条目
4. 可选：集成 small embedding model（multilingual-e5-small ~118MB 或 bge-small-zh ~24MB），在 `catalog.json` 中存储 float32 嵌入向量，query 时做 BM25 + cosine similarity 混合评分

**迁移复杂度：** 中 — 纯 JSON + Python，零额外运行时依赖

**投入产出比：** 高

**优先级：** P1；触发条件：真实检索出现稳定的同义词或上下位概念漏召回

### Issue 9: [架构演进] Generator-Verifier 复合 AI 模式

**Labels:** `enhancement`, `architecture`, `agent-gateway`

**Body:**

在 AI 答案生成和教师审核之间插入验证器步骤，使用独立的（通常更小/更便宜）模型检查生成质量。

**解决的问题：** 答案验证仅做结构性检查（见 Issue 6），教师承担全部语义审核

**方案要点：**
1. 定义验证 Schema：`answer_validity`（pass/fail/uncertain）、`issues[]`（error/warning/info 严重级别）、具体步骤引用
2. 实现 `verify_answer` 函数，发送题目文本 + 生成答案 + physics-model.json（如有）给验证 provider
3. 复用现有 agent gateway 的 provider 链（验证可用更便宜的模型或本地 SLM）
4. 验证报告存储为条目内的 `verification.json`
5. 与 HITL 风险分流联动：验证通过可降低队列风险但不自动批准；验证失败标记具体问题；不确定项带验证备注交给教师

**迁移复杂度：** 中

**投入产出比：** 中高

**优先级：** P1；触发条件：验证 Schema、失败回退和教师呈现方式完成评审

### Issue 10: [架构演进] MCP 协议标准化工具接口

**Labels:** `enhancement`, `architecture`, `teacher-console`

**Body:**

将现有 OCR、知识库、物理仿真、答案生成等能力包装为 MCP Server（stdio 传输），使每个工具成为自描述、可独立测试、带统一校验的标准化单元。

**解决的问题：** Skill 编排使用临时函数调用（见 Issue 5）

**方案要点：**
1. 从知识库开始（`kb.py` 已有清晰的函数接口），使用 FastMCP Python 库的 `@mcp.tool()` 装饰器包装
2. 以 stdio 子进程方式运行 MCP Server，零网络配置
3. 在教师工作台中创建 MCP Client 实例进行工具发现
4. 逐步迁移 OCR 服务、物理仿真构建器、答案验证器等
5. 利用 Resources 原语提供知识库条目只读访问，Prompts 原语提供审核工作流模板
6. 工具注解（destructiveHint、readOnlyHint）提供自动安全护栏

**迁移复杂度：** 中

**投入产出比：** 中高

**优先级：** P2；触发条件：出现第三类以上外部工具接入，现有函数调用开始造成重复适配

### Issue 12: [架构演进] 本地 SLM 离线 AI 推理降级

**Labels:** `enhancement`, `architecture`, `agent-gateway`

**Body:**

为现有 Agent Gateway 增加 Ollama 等本地小模型的自动发现、配置和质量提示，继续复用已经存在的结构化 provider 协议。

**解决的问题：** 本地 SLM 能力可接入但不够开箱即用，未预配时断网无法生成内容（见 Issue 3）

**方案要点：**

3. 配置成功的本机 provider 在 auto 链中优先于远程 CLI，并如实报告数据位置
4. 推荐模型：Qwen2.5-3B-Instruct（中文+物理）、Phi-4-mini（推理）
5. UI 警告“本地模型生成，仍需教师复核”，支持稍后用其他 provider 重新运行
6. 本地模型做默认验证器（验证比生成容易），云端做生成

**迁移复杂度：** 中

**投入产出比：** 中

**优先级：** P2；触发条件：完成目标学校设备与弱网环境评估，并确定可运行的本地模型规格

### Issue 13: [架构演进] BKT 知识追踪 + 薄弱点诊断

**Labels:** `enhancement`, `architecture`, `knowledge-base`

**Body:**

为每个物理概念添加基于简化贝叶斯知识追踪（BKT）的掌握度跟踪，实现量化薄弱点诊断。

**解决的问题：** 当前薄弱点分析依赖 error_types 标签和教师人工解读，无法区分"真不会"和"粗心错"

**方案要点：**
1. 建立概念-题目映射（依赖 Issue 8 的概念本体）
2. 为每个物理概念初始化 BKT 参数：P(L₀)=0.3（初始掌握概率）、P(G)=0.2（猜测概率）、P(S)=0.1（失误概率）
3. 每次复习后（正确/部分正确/错误），用贝叶斯定理更新掌握概率
4. 新增薄弱点 API 端点：返回按 P(掌握) 升序排列的概念，过滤到有 ≥3 次观察的
5. 教师工作台可视化：概念掌握度热力图
6. 复习调度联动：低掌握度概念获得更高频复习

**迁移复杂度：** 中 — 纯数学 ~100 行 Python，但需要概念本体（Issue 8）作为前提

**投入产出比：** 中

**优先级：** P2；触发条件：概念本体可用，且目标概念至少积累 3 次有效学习观察

### Issue 14: [架构演进] MiniRAG 跨条目主题问答（远期储备）

**Labels:** `enhancement`, `architecture`, `knowledge-base`, `future`

**Body:**

当知识库增长到 100+ 条目后，引入 MiniRAG（港大 2025，SLM 优化的轻量 GraphRAG）做跨条目主题问答。

**解决的问题：** 本体增强 BM25 在 100+ 条目时仍难以处理跨条目聚合查询（如"这个学生在电磁学哪个具体概念上最容易出错？"）

**方案要点：**
1. 前提条件：100+ 已审核条目 + 足够的使用数据
2. 使用 MiniRAG 而非 Microsoft GraphRAG（MiniRAG 支持 1.5B 模型 vs 微软需 7B+）
3. 相似图社区检测（比 LLM-based Leiden 社区检测更省计算）
4. 支持增量更新（vs 微软需要全文重建）
5. 离线批处理（夜间/空闲时运行索引）
6. 使用物理概念本体作为种子实体减少抽取错误
7. 暴露 Global Search（MapReduce 社区摘要）做主题查询，Local Search（实体锚定）做概念查找

**迁移复杂度：** 高

**投入产出比：** 低（在 <100 条目规模下）

**优先级：** P3；触发条件：达到 100+ 个已审核条目并出现跨条目聚合查询需求
