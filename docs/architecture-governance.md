# 悟理架构治理协议

这份文档用于在新增功能、删减入口、修复跨模块问题或准备竞赛展示前，借助 `graphify` 做项目级治理。它不是新的业务 Skill，也不替代现有 Skill；它把 `graphify` 的项目地图能力转化为悟理自己的架构判断流程。

一句话定位：

```text
graphify 负责看见结构；本协议负责基于结构做取舍。
```

## 什么时候使用

遇到以下问题时，先按本协议走一遍，而不是直接改代码：

- “这个功能应该放在哪个模块？”
- “这个按钮是不是多余？”
- “教师端、学生端、Skill、Agent Gateway 有没有职责混淆？”
- “这次改动会影响哪些文档、测试和交付流程？”
- “项目能不能作为一个完整系统讲清楚？”
- “某个能力做了，但用户为什么还是不知道怎么用？”

不需要使用本协议的情况：

- 单道题 OCR、入库、导出、复习；
- 单个物理仿真模型的构建或修复；
- 会话结束时的普通文档同步；
- 只读侦查下一步优先级。

这些分别交给 `manage-student-error-library`、`build-physics-simulator`、`neat-freak` 和 `scout`。

## 与现有工具的边界

| 对象 | 职责 | 本协议如何使用它 |
|---|---|---|
| `graphify` | 生成/查询项目知识图谱，发现文件、概念、调用和文档关系 | 作为结构证据来源 |
| `scout` | 阶段开局，只读侦查风险与优先级 | 可作为治理前的 P0/P1 排序输入 |
| `neat-freak` | 阶段收尾，同步文档与规则 | 治理改动完成后用于收尾审计 |
| `manage-student-error-library` | 错题生命周期总控 | 不把治理逻辑塞进业务流程 |
| `build-physics-simulator` | 物理模型与交互仿真专家 | 不承担答案、OCR、PDF 或知识库职责 |
| `teacher-console/agent_gateway.py` | Agent provider、候选隔离、模型路由、安全提升 | provider 细节不得重新散落进 `server.py` |

## 标准流程

### 1. 先问清楚治理对象

把需求归入一种类型：

| 类型 | 典型问题 | 输出 |
|---|---|---|
| 功能归位 | 新功能放哪里 | 模块归属与不应修改的边界 |
| 复杂度删减 | 哪些按钮、配置、文案会误导用户 | 删除/合并/隐藏建议 |
| 影响分析 | 改一个点会牵动什么 | 文件、测试、文档清单 |
| 流程审计 | 上传到交付是否闭环 | 生命周期断点与门禁检查 |
| 展示叙事 | 项目如何讲给评委 | 架构图、演示脚本、创新点证据 |

### 2. 用 graphify 查结构证据

优先查询已有图谱：

```bash
graphify query "这个改动会影响哪些模块、文档和测试？"
```

如果近期做过大量结构调整，再更新图谱：

```bash
graphify update .
```

查询时不要只问“优化项目”，要带上明确边界，例如：

```bash
graphify query "教师端模型设置 UI、agent_gateway.py、model-registry 文档之间是否有分叉？"
graphify query "manage-student-error-library 和 build-physics-simulator 的职责边界在哪里？有没有重叠风险？"
graphify query "公开学生端发布流程中哪些文件允许公开，哪些文件必须留在本地？"
```

### 3. 做悟理专用边界检查

每次治理判断至少检查这些边界：

- `manage-student-error-library` 只能总控错题生命周期，不应内置具体仿真渲染逻辑；
- `build-physics-simulator` 只拥有 `physics-model.json`、物理事件、轨迹、交互 HTML/ZIP，不写 OCR、答案、PDF、知识库；
- `teacher-console/server.py` 只负责本地 HTTP、生命周期动作和任务提交，不重新承载 provider 参数细节；
- `teacher-console/agent_gateway.py` 是 Agent provider、候选隔离、模型路由和安全提升的统一入口；
- Agent 永远不能批准题干、批准答案、批准可视化、finish 或发布学生端；
- `student-site/` 是只读公开站，不能引用原始上传、教师版解析、条目内部 JSON、本地绝对路径或本地 API；
- LiteLLM 是高级外部网关方案，不是悟理默认启用能力；教师端不应暗示它开箱即用；
- “经济 / 自动 / 深度 / 自定义”是教师可理解的模式，真实模型细节只在设置页必要处暴露；
- 标准解析不自动生成交互仿真；可视化页面保留入口，教师明确请求后才生成。

### 4. 用“用户心智”审 UI

任何新增入口都要回答：

1. 教师看到它时，能不能知道下一步该做什么？
2. 它是否暗示了一个并未默认启用的能力？
3. 它是否把开发者配置问题暴露给普通教师？
4. 它是否能被“设置页 + 简短说明 + 测试按钮”解释清楚？
5. 如果失败，错误信息能不能告诉教师可执行动作？

不满足时优先：

```text
删除入口 > 合并入口 > 移到高级设置 > 加长说明
```

## 常用治理查询

### 模型与 Agent

```bash
graphify query "教师端模型模式、model-registry、agent_gateway.py 和 openai-compatible adapter 的关系是什么？"
```

检查重点：

- UI 的模式说明是否与 `resolve_model_id_for_task` 一致；
- `economy` / `expert` 是否只是默认映射，而不是题目难度智能判断；
- API Key 是否只保存在本地忽略文件，不回显、不入任务 JSON；
- 未测试模型是否不会被默认调用；
- provider 参数是否仍统一经 Gateway 传递。

### 错题生命周期

```bash
graphify query "一题从上传到 delivered 经过哪些脚本、状态文件、复核门禁和输出产物？"
```

检查重点：

- `delivery-manifest.json` 是否仍是最终完成真源；
- 题干、答案、可视化、公开发布是否各有独立复核；
- 修改受保护文件后旧批准是否失效；
- `kb.py rebuild` 是否在必要修改后执行，避免旧方法污染检索。

### 可视化边界

```bash
graphify query "可视化生成涉及哪些 physics-model、HTML、ZIP、运行时检查和教师复核文件？"
```

检查重点：

- 新题型是否需要扩展 `model_type`、renderer、schema、校验和测试；
- 答案与 HTML 是否共享同一个 `physics-model.json`；
- `finish` 是否复制已批准字节，而不是临时重建；
- 不适合确定性仿真时是否返回 `unsupported`，而不是生成误导动画。

### 学生端发布

```bash
graphify query "student-site 发布流程中哪些文件允许公开，哪些文件必须留在本地？"
```

检查重点：

- 公开 PDF 是否由脱敏 Markdown 和公开题图重新生成；
- 学生端 viewer 是否只读相对路径资源；
- 原始题图、教师版、内部 JSON、绝对路径是否被阻断；
- GitHub 推送是否仍是明确人工动作。

```md
## 用户困惑点

一句话描述用户为什么会困惑。

## 当前实现

代码/文档/UI 分别在哪里。

## 是否值得保留

- 保留价值：
- 增加成本：
- 推荐动作：删除 / 合并 / 移到高级设置 / 改文案

## 改动后用户路径

用 3-5 步描述教师实际怎么完成任务。
```

## 是否升级成 Skill

暂时不把本协议升级成 Skill。只有当它被重复使用并满足以下条件时，再创建 `wuli-architecture-governor`：

- 至少 3 次独立治理任务都复用了同一套检查项；
- 每次都需要固定输出报告；
- 手工执行成本高于维护一个 Skill 的成本；
- 与 `graphify`、`scout`、`neat-freak` 的职责边界稳定清晰。

在此之前，本协议保持为文档。这样不会为了治理工具再制造一个需要治理的新对象。
